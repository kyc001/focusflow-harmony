\chapter{引言}

\section{研究背景}

移动互联网改变了大学生的学习组织方式。课程资料不再只来自教材，还包括课件、实验文档、网页链接、课堂笔记和复习清单等多种来源\cite{crompton2017use}。这些资料常分散在备忘录、网盘、浏览器收藏夹和聊天记录中，导致学生在真正执行学习任务时需要反复寻找上下文，任务拆解、专注执行和复盘反馈也难以形成连续链路。现有工具通常只覆盖局部场景：待办工具擅长记录任务，番茄钟工具擅长维持注意力，笔记工具擅长沉淀材料，而通用 AI 助手擅长内容生成，却很少把资料、任务、专注和复盘组织成统一的移动端工作流\cite{ducate2016mlearning}。

近年来，大语言模型为学习工具提供了资料提炼、行动建议和复习规划等能力\cite{kasneci2023chatgpt}。但 AI 服务往往依赖网络和 API 密钥，若将其作为核心功能前提，会削弱移动应用在离线或弱网环境下的可用性。与此同时，HarmonyOS 与 ArkTS 为国产移动应用开发提供了新的平台能力\cite{harmonyos2024}。因此，本文以“移动应用开发”课程项目为背景，设计并实现一款面向大学生的 HarmonyOS 学习闭环应用“知序”（ZhiXu），探索本地优先架构与可选 AI 增强在学习管理场景中的结合。

\section{研究目标与内容}

本文目标是在毕业论文格式下，完整描述“知序”的需求、设计、实现与验证。系统以端侧 RDB 为核心数据源，保证任务管理、资料保存、番茄记录和复盘统计在离线环境下可用；AI 作为增强层，仅在用户启用并配置密钥后参与资料提炼和学习建议生成；Spring Boot 后端保留为可选同步层，用于展示全栈扩展能力，而不是移动端运行的必要条件。

围绕上述目标，本文主要完成以下工作：第一，设计项目、任务、番茄记录和学习资料四类核心实体，并基于 HarmonyOS RDB 实现本地持久化；第二，构建今日、任务、资料、专注和复盘五个核心页面，形成从资料沉淀到任务执行再到统计反馈的学习闭环；第三，设计远程 AI 请求与显式状态降级机制，避免在未配置或请求失败时伪造本地 AI 结果；第四，实现 FocusAbility 独立专注窗口、ArkWeb 复盘图表和服务卡片等 HarmonyOS 平台能力；第五，对系统进行功能测试、构建验证和兼容性观察。

\section{论文组织}

全文共七章。第一章介绍研究背景和目标；第二章综述相关技术；第三章分析系统需求；第四章给出总体架构、数据模型、AI 模块和后端边界设计；第五章说明关键模块实现；第六章进行测试与验证；第七章总结工作并展望后续优化。


\chapter{相关工作与技术基础}

\section{HarmonyOS 与 ArkTS}

HarmonyOS 是面向多设备协同的操作系统，应用采用 Stage 模型组织 Ability 生命周期。ArkTS 在 TypeScript 基础上增强了声明式 UI 能力，开发者通过状态驱动界面更新，适合构建结构化的移动应用页面\cite{zhang2023arkui}。在数据持久化方面，HarmonyOS 提供关系型数据库 RDB 和轻量级 Preferences。前者适合保存项目、任务、番茄记录等结构化数据，后者适合保存登录态、计时器偏好和 AI 配置等运行时设置。本文系统正是基于这两类端侧能力建立本地优先的数据基础。

\section{移动学习与番茄工作法}

移动学习强调在任意时间、任意地点利用移动设备完成学习活动\cite{shadiev2017mlearningreview}。现有移动学习工具常以“单点能力”为核心，例如任务清单、计时器或笔记库。对于大学课程学习而言，仅有单点能力仍不足以支撑完整流程，因为学习资料需要转化为可执行任务，任务需要通过专注过程完成，完成情况又需要被统计和复盘。

番茄工作法将学习时间切分为固定长度的专注区间，并在区间之间安排休息\cite{cirillo2018pomodoro}。该方法适合课程复习、实验编程和论文阅读等需要持续注意力的任务。本文没有把番茄钟设计为孤立计时器，而是将每次完成或中断都写入本地数据库，使专注过程能够反向支撑复盘统计。

\section{AI 辅助学习与本地优先思想}

AI 辅助学习可以帮助学生从资料中提取摘要、要点和下一步行动\cite{wang2024aitutor}。但移动端应用需要处理网络波动、密钥缺失和生成内容不稳定等问题。本地优先思想主张核心数据和关键操作应优先在端侧完成，远程服务只作为增强或同步能力存在\cite{turnbull2023localfirst}。因此，“知序”将任务、资料、番茄和复盘均放在本地闭环中，远程 AI 只在可用时生成建议，并在不可用时返回明确状态提示。

\section{本章小结}

本章介绍了 HarmonyOS、ArkTS、移动学习、番茄工作法和 AI 辅助学习的相关基础。上述技术共同决定了本文系统的设计方向：以端侧能力保证确定性流程，以 AI 增强提升资料处理效率，以移动端交互整合学习闭环。


\chapter{系统需求分析}

\section{系统概述}

“知序”面向使用 HarmonyOS 设备的大学生，典型场景包括课堂笔记整理、实验任务拆解、考试复习和课程项目推进。系统定位为本地优先的 AI 学习工作流工具，核心目标不是替代所有学习软件，而是将“资料整理、任务分解、番茄执行、复盘反馈”集中到一个轻量但连续的移动端流程中。

\section{非功能需求}

系统需满足以下非功能要求：离线可用，核心功能不依赖后端和 AI；数据持久化，应用重启后任务、资料和番茄记录不丢失；安全性，API 密钥只作为运行时配置保存在 Preferences 中，不写入源码或报告；可维护性，页面不直接操作数据库，统一通过 Store 和服务层完成；兼容性，面向 HarmonyOS 6.0 及以上设备，并为手机、平板和二合一设备保留响应式布局空间。

\section{典型用例分析}

从用户视角看，系统最重要的用例是“从资料到行动”。学生在资料页录入课堂笔记、课程链接或本地文件 URI，系统先将其作为 StudyResource 写入本地数据库；若用户配置了远程 AI，则可以进一步生成摘要和关键要点；随后用户可将资料提炼结果转化为专注任务，任务进入今日页和任务页，并在番茄专注完成后形成可复盘的时间记录。该用例贯穿资料、任务、专注和复盘四个模块，是本文系统区别于普通待办或普通笔记应用的核心链路。

另一个重要用例是“离线学习记录”。当用户没有网络、后端未启动或 AI 未配置时，系统仍应允许创建任务、保存资料、开始番茄、记录中断原因和查看本地统计。此时远程 AI 卡片只提供配置状态提示，后端同步入口也不阻断本地写入。该用例约束了后续架构设计：所有核心数据必须以端侧 RDB 为可信来源，远程能力只能在本地成功之后补充，而不能反向决定本地流程是否成立。

\section{本章小结}

本章明确了系统的目标用户、核心功能和非功能约束。需求分析表明，“知序”的关键不是堆叠页面数量，而是保证资料、任务、专注和复盘之间的数据链路稳定、可解释且可离线运行。

\section{AI 模块设计}

AI 模块采用远程增强与显式状态降级策略。AI 设置保存在 focus\_ai Preferences 中，包含 enabled、endpoint、model 和 apiKey。用户保存设置时，系统归一化 endpoint 和 model；若输入框中的密钥为空，则保留已保存密钥，且页面不回显明文密钥。远程请求兼容 Chat Completions 风格接口，服务层将响应解析为 AiCoachAdvice 或 ResourceInsight。

算法 \ref{alg:remote-ai} 描述了 AI 请求的核心流程。若远程模式关闭、密钥缺失、HTTP 状态异常、响应为空或格式不可解析，系统返回配置或服务状态提示，不生成本地替代结果，也不写回资料摘要。这样可以保证用户清楚地区分“远程模型生成内容”和“本地状态提示”。

\begin{algorithm}[htbp]
\caption{远程 AI 请求与状态降级算法}
\label{alg:remote-ai}
\begin{algorithmic}[1]
\Require AI 设置 $settings$，本地上下文 $context$
\Ensure AI 结果 $result$ 或状态提示 $status$
\If{$settings.enabled = false$ \textbf{or} $settings.apiKey$ 为空}
  \State \Return 配置状态提示
\EndIf
\State $response \leftarrow \text{HTTP\_POST}(endpoint, context, apiKey)$
\If{$response.code < 200$ \textbf{or} $response.code \geq 300$}
  \State \Return 远程服务状态提示
\EndIf
\State $content \leftarrow \text{extractChoiceContent}(response)$
\If{$content$ 为空或不可解析}
  \State \Return 响应解析状态提示
\EndIf
\State \Return $\text{parseStructuredLines}(content)$
\end{algorithmic}
\end{algorithm}

\section{后端与同步边界}

后端使用 Spring Boot 3.2.0、MyBatis 和 MySQL 风格 schema，提供用户、项目、任务、番茄和同步接口。同步设计采用 updated\_at 时间戳进行增量 pull/push，番茄记录通过 client\_request\_id 实现幂等插入。需要说明的是，当前移动端尚未将同步 UI 完全产品化，因此本文将后端定位为课程全栈展示和未来多端备份基础，而非已完成的实时多端同步系统。

\section{本章小结}

本章给出了“知序”的总体架构和关键设计。端侧 RDB 保证核心闭环，Preferences 保存运行时配置，FocusStore 统一页面写入路径，远程 AI 和后端同步均作为可选增强存在。这些设计共同支撑了系统的离线可用性和可维护性。

\section{核心页面实现}

Index 页面承载今日、任务、资料、专注和复盘五个工作区。页面通过 @State 保存任务、资料、计时器、AI 配置和当前选中项；aboutToAppear() 中按顺序初始化本地数据库、设置服务、Store、登录仓库和 AI 设置，随后调用 syncFromStore() 同步页面状态。今日页的快速添加任务只调用 focusStore.addTask()，不直接修改 this.tasks，因此离线状态下仍能正常新增任务并在重启后保留。

任务管理模块实现了搜索、筛选、四象限视图和回收站。filteredTasks() 会按项目、优先级、关键词和回收站状态组合过滤，关键词覆盖标题、备注和标签。toggleTask()、deleteTask() 和 restoreTask() 均通过 Store 修改 RDB，再刷新页面状态。三态状态机避免了误删数据，也为后续同步保留了软删除语义。

\section{学习资料库实现}

资料库是系统的主要特色。StudyResource 模型包含 title、sourceType、projectId、linkedTaskId、content、summary、keyPoints、tags、status、createdAt、updatedAt 和 reviewDueAt 等字段。手动录入时，页面校验标题和正文后调用 addStudyResource() 写入本地表；文件导入时，DocumentViewPicker 只保存文件 URI 和用户备注，不复制二进制文件，也不解析 PDF、DOCX 或 PPTX 全文。这样既展示了 HarmonyOS 文件选择能力，又控制了课程项目阶段的权限和依赖复杂度。

AI 提炼由 extractResource() 触发。页面先检查远程 AI 是否启用且存在密钥；不满足条件时只显示状态，不写入摘要。远程返回成功后，FocusAiCoachService 将响应解析为 ResourceInsight，Store 再更新 summary 和 keyPoints。用户可基于提炼结果调用 createTaskFromResource() 创建普通 FocusTask，并把资料的 linked\_task\_id 回写为新任务 id，从而建立资料与任务的闭环关系。

\section{番茄专注与原生能力实现}

番茄专注支持内嵌模式和独立窗口模式。内嵌模式在 Index 中以 Overlay 展示计时器；独立模式通过 FocusAbility 启动 FocusSolo 页面。Index 将 taskId 和 taskTitle 放入 Want parameters，FocusAbility 在 onCreate() 和 onNewWant() 中均解析参数并写入 AppStorage，FocusSolo 通过 @StorageProp 读取当前任务。这样即使 FocusAbility 以 singleton 方式热启动，也能切换到新的专注对象。

计时器使用 setInterval 每秒更新剩余时间，完成时通过 FocusNativeServices 发送本地通知，并调用 focusStore.recordPomodoro() 写入 pomodoros 表。后台运行时，系统尽量请求 suspend delay 保护专注过程；若平台能力不可用，计时器仍保持本地可用。FocusSolo 会从 Preferences 读取用户保存的专注时长，失败时退回默认 25 分钟。

\section{复盘统计与扩展能力实现}

复盘统计数据全部来自 FocusStore，而不是来自种子数据或 H5 页面假数据。Store 根据 tasks 和 pomodoros 快照计算 completedPomodoros()、interruptedPomodoros()、totalFocusMinutes()、weekFocusPoints()、projectFocusPoints() 和 streakDays()。ArkWeb 加载 rawfile/charts/index.html，通过 nativeBridge 接收统计 JSON 并使用 Canvas 绘制图表；当桥接不可用时，H5 页面提供 fallbackStats() 便于预览。

项目还实现了 EntryFormAbility 与 FocusCard 服务卡片，用于展示今日概览和番茄进度。当前卡片采用静态演示数据，主要体现 ExtensionAbility 与卡片布局能力，后续可进一步接入真实 Store 快照。

\section{关键技术问题}

实现过程中主要处理了四类问题。第一，跨层数据一致性：所有核心写操作统一走 FocusStore，避免页面数组与 RDB 分裂。第二，AI 不稳定性：失败时返回明确状态，不伪造本地 AI 结果。第三，产品更名控制：用户可见名称统一为“知序”，内部 Focus 前缀类名保持稳定，减少重命名风险。第四，文件导入范围控制：当前只保存 URI 和备注，文档正文解析、OCR 和权限扩展留作后续工作。

\section{本章小结}

本章说明了“知序”的主要实现过程。系统在页面、Store、RDB、Preferences、Ability 和 ArkWeb 之间建立了稳定的数据协作方式，并围绕资料库、番茄专注和复盘统计完成了课程项目要求的核心功能。

\chapter{总结与展望}

\section{工作总结}

本文完成了 HarmonyOS 本地优先学习闭环应用“知序”的设计与实现。系统以 RDB 保存项目、任务、番茄记录和学习资料，以 Preferences 保存设置、登录态和 AI 配置，通过 FocusStore 统一本地写入路径，完成了今日、任务、资料、专注和复盘五个核心页面。AI 模块采用远程增强和显式状态降级设计，在配置可用时提供学习建议和资料提炼，在不可用时不影响本地流程。后端保留为可选同步层，体现了全栈扩展能力。

\section{主要特色}

本项目的特色主要体现在三个方面：第一，本地优先架构使核心学习流程不依赖网络、后端或 AI；第二，资料库与任务系统打通，使学习资料能够转化为可执行的专注任务；第三，HarmonyOS 平台能力被用于独立专注窗口、服务卡片、原生通知、后台保护和 ArkWeb 图表展示，增强了课程项目的移动端完整性。

\section{不足与展望}

系统仍存在不足：文件导入目前只保存 URI 和备注，尚未实现 OCR 与文档正文解析；后端同步 UI 尚未产品化，冲突处理和同步状态提示仍需完善；AI 功能依赖外部 API 服务，缺少本地模型或缓存策略。后续可继续扩展文档解析、错题整理、学习周报、多端备份、真机兼容性测试和无障碍适配，使应用更接近可长期使用的成熟学习工具。
