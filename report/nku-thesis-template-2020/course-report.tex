\chapter{序论}

\section{研究背景和意义}

移动端学习工具已经成为大学生管理课程任务的重要入口。课程学习往往不是单一的待办清单，而是由课件、笔记、网页资料、论文摘录、实验报告和复习计划共同组成。实际使用中，学生经常把资料散落在聊天记录、浏览器收藏、文档目录和便签软件中，导致资料难以复用，任务难以拆解，复习节奏也缺少数据反馈。

已有工具大致可以分为三类：第一类是通用待办工具，适合记录任务，但难以体现学习资料和专注过程；第二类是番茄钟工具，能够帮助用户保持注意力，但缺少任务上下文和复盘分析；第三类是笔记工具，擅长沉淀材料，但通常不会直接推动下一次行动。随着大语言模型能力增强，学习工具可以进一步把资料内容转化为摘要、要点和下一步任务，从而减少学生从“看资料”到“开始行动”之间的摩擦。

因此，本项目将学习资料、任务管理、番茄专注和复盘统计组合成一个本地优先的移动应用。其意义主要体现在三个方面：一是通过 RDB 和 Preferences 保证离线可用，适合课堂演示和日常使用；二是通过资料库与 AI 提炼功能提升任务生成能力，使应用不只是展示界面；三是通过可选后端展示服务器和数据库连接能力，覆盖移动端、数据库、网络服务等课程知识点。

\section{研究内容}

本文围绕“知序”应用展开，重点研究并实现以下内容：

\begin{enumerate}
  \item 设计本地优先的数据模型，覆盖项目、任务、番茄记录和学习资料。
  \item 构建包含今日、任务、资料、专注、复盘五个核心页面的 HarmonyOS 应用。
  \item 引入 AI 学习助手能力，在没有网络或密钥时使用本地规则，在配置运行时模型接口后使用远程模型增强。
  \item 保留 Spring Boot + MyBatis + MySQL 后端，作为可选同步接口和课程全栈展示能力。
  \item 通过构建验证、数据持久化设计和报告整理，说明系统的实现效果与可扩展方向。
\end{enumerate}

\section{本文组织结构}

第一章介绍项目背景、研究意义和本文结构。第二章比较开发语言、开发工具、前端框架、数据库和后端框架的选型依据。第三章从总体架构、MVVM 分层、数据库设计、AI 模块和网络模块等方面说明系统设计。第四章描述系统实现效果、关键问题和解决方法。第五章总结本文工作，并提出后续优化方向。

\chapter{工具选型}

\section{开发语言与平台选择}

本项目面向 HarmonyOS 移动端，首选 ArkTS 作为端侧开发语言。与 Java、Kotlin、JavaScript 等方案相比，ArkTS 与 ArkUI 在 HarmonyOS 工程中集成度更高，能够直接使用 RDB、Preferences、Ability、FormExtensionAbility、ArkWeb、通知和后台任务等平台能力。JavaScript 的跨平台生态成熟，但在 HarmonyOS 原生能力调用和类型约束方面不如 ArkTS 稳定；Kotlin 更适合 Android 原生开发，不适合作为本项目的主技术路线。

因此，端侧选择 ArkTS + ArkUI，后端选择 Java + Spring Boot。这样的组合既能体现鸿蒙端侧开发特点，也能展示服务器端和数据库连接能力。

\section{开发工具选择}

DevEco Studio 是 HarmonyOS 项目的官方开发工具，提供工程创建、ArkTS 编译、资源管理、预览、签名和 HAP 打包等能力。与 Eclipse、IntelliJ IDEA 等通用 IDE 相比，DevEco Studio 对鸿蒙 SDK、hvigor 构建和真机调试支持更直接。后端部分使用 Maven 管理 Spring Boot 项目，可以在 IDEA 或命令行中构建。

本项目使用 DevEco Studio 工程结构进行端侧开发，并通过命令行 hvigor 完成 HAP 构建验证；报告使用本 LaTeX 模板生成，答辩材料使用 Typst 生成，便于自动目录和 PDF 输出。

\section{框架与数据库选择}

前端框架选择 ArkUI 声明式 UI。ArkUI 能够以组件化方式描述页面结构，适合构建多页面、多状态的移动端应用。本地数据库选择 HarmonyOS RDB，用于保存结构化学习数据；轻量配置选择 Preferences，用于保存登录态、计时设置和 AI 运行时配置。服务端选择 Spring Boot + MyBatis + MySQL，原因是其工程结构清晰、接口开发效率高、数据库映射直观，适合课程项目演示。

\chapter{系统设计}

\section{总体设计}

“知序”采用本地优先的移动应用架构。应用启动后先初始化本地数据库和偏好设置，所有核心学习数据先写入端侧 RDB，再由页面状态读取并展示。后端同步作为可选能力存在，不作为用户正常使用的前提。这样即使没有网络、没有服务器或没有配置 AI 密钥，用户仍然可以完成任务记录、番茄专注和复盘统计。

系统采用 MVVM 思路组织。模型层负责定义项目、任务、番茄记录、学习资料、AI 建议和设置等数据结构，并通过 RDB 表进行持久化。视图层负责展示今日、任务、资料、专注、复盘等页面，处理表单输入、筛选、按钮动作和状态提示。控制层负责连接视图和模型，包括本地数据仓库、状态门面、AI 服务、设置服务、通知和后台任务服务。页面不直接操作数据库，而是通过统一的 Store 写入数据，再刷新本地快照。

\section{功能模块设计}

系统包含五个核心页面。新增“资料”页面后，系统形成了更完整的学习闭环：用户先沉淀资料，再提炼摘要和要点，然后生成任务，最后进入专注并在复盘页面查看结果。

\section{AI 模块设计}

AI 模块被设计为增强层，而不是系统运行的前提。其输入来自本地任务、项目、番茄记录或资料内容，输出为学习建议、摘要、要点和下一步任务建议。AI 模块包含两种模式。

本地规则模式不依赖网络和密钥。系统根据任务优先级、截止时间、中断次数、资料长度和资料类型生成建议，保证任何情况下都有可用结果。远程模型模式读取用户在运行时输入的接口地址、模型名和 API Key。系统不会把用户提供的密钥写入源码或报告。远程请求失败、返回为空或格式不符合要求时，系统自动回退到本地规则。

\section{网络与后端设计}

后端目录保留 Spring Boot 服务，包含用户、项目、任务、番茄、统计和同步接口。后端通过 MyBatis 映射 MySQL 数据表，适合演示“服务器连接数据库”的课程要求。端侧仍然坚持本地优先原则：本地写入成功是核心路径，后端同步是附加路径。这样既能满足离线使用，也能在答辩时展示前后端联调与数据库设计能力。

\section{代码规范与可维护性设计}

系统在实现上遵循以下规范：页面状态不直接修改数据库；所有核心写操作通过 Store 完成；ResultSet 使用后及时关闭；密钥仅作为运行时配置保存，不出现在源码、报告和任务文档中；新增功能尽量复用现有模型、服务和 UI 样式，避免大范围重命名造成风险。内部类名保留 Focus 前缀，用户可见产品名统一为“知序”，兼顾稳定性和产品表达。

\chapter{系统实现}

\section{运行效果}

应用完成后形成五个真实页面，每个页面都有可操作功能。今日页展示学习状态、快速添加任务和 AI 建议；任务页支持搜索、筛选、删除和恢复；资料页支持新增资料、提炼摘要、生成任务和归档；专注页支持番茄计时、中断记录和通知；复盘页展示本地统计、ArkWeb 图表和学习森林。

本次迭代将产品名称调整为“知序”，副标题定位为“AI Study Flow”。可见界面包括主登录页、今日页标题、服务卡片和独立专注页面均使用新的产品表达。内部文件和类名保持不变，避免破坏 Ability 路由和构建配置。

\section{学习资料库实现}

资料库是本次成熟化迭代的重点。用户可以录入笔记、网页链接说明、课件摘录和 PDF/论文摘录，并将资料关联到具体项目。资料保存后写入本地 RDB，页面从 Store 刷新数据并展示列表。资料卡片展示类型、所属项目、摘要、要点和关联任务状态，用户可以选择资料、提炼资料、归档资料或恢复资料。

资料页不是静态展示页，而是参与任务闭环的功能页。用户完成资料提炼后，可以一键生成专注任务。生成的任务会带有资料摘要和要点，进入任务页与专注页后继续使用原有番茄闭环。

\section{AI 提炼实现}

系统为资料库增加了本地提炼和远程提炼两条路径。本地提炼会根据资料内容切分句子，生成摘要、三个要点、建议任务和预计专注分钟数。远程提炼复用已有 AI 设置，读取运行时保存的接口地址、模型和密钥，要求模型返回摘要、要点、任务和分钟数。若远程模式未启用、密钥为空、网络失败或响应不可解析，系统会显示本地兜底结果。

这种设计使 AI 功能既能用于演示创新点，又不会让应用依赖外部服务。对课程项目而言，AI 是加分项，本地可用性仍然是主线。

\section{本地持久化实现}

系统数据库版本升级后新增学习资料表。启动时应用创建或迁移表结构，并在空表时写入示例资料。任务、项目、番茄和资料均通过统一数据库服务读取和写入。页面通过 Store 获取快照，写操作完成后刷新 Store，再同步到 ArkUI 状态。

这一实现解决了只在内存中展示数据的问题。用户新增任务、完成番茄或保存资料后，数据能够在应用重启后保留，复盘统计也来自本地记录，而不是静态样例。

\section{关键问题与解决方法}

第一个问题是资料库跨越模型、数据库、状态和页面四层，容易出现某层遗漏。解决方法是先定义 StudyResource 模型，再新增 RDB 表和数据库方法，随后补 Store 方法，最后接入 ArkUI 页面，保持数据流单向清晰。

第二个问题是 AI 远程请求具有不稳定性。解决方法是把本地规则作为默认结果，远程请求只负责增强；无密钥、失败或响应异常时都回退到本地结果，并在页面显示来源。

第三个问题是产品改名可能牵动 Ability、资源和已有代码。解决方法是只更新用户可见文案，内部类名、文件名和路由保持稳定。这样既能让产品名称更成熟，又不会引入大范围重构风险。

第四个问题是报告要求第三章和第四章不能贴代码。本文在这两章使用文字、表格和结构说明描述实现，不展示源码片段，保证读者按系统设计和实现效果理解项目。

\section{验证结果}

应用端已经使用 hvigor 执行 HAP 构建，ArkTS 编译、资源处理、打包和签名流程均通过。构建过程中存在若干历史警告，例如 getContext 弃用提示、后台任务能力兼容性提示和已有 Preferences 异常处理提示，但没有阻塞本次功能构建。报告使用南开大学本科生毕业论文 LaTeX 模板生成，答辩 slides 使用 Typst 编译为 PDF，便于提交和展示。

\chapter{总结与展望}

\section{总结}

本文完成了“知序”学习管理应用的设计与实现。系统以 HarmonyOS 端侧能力为基础，完成今日、任务、资料、专注、复盘五个真实页面，使用 RDB 和 Preferences 实现本地优先的数据闭环，并保留 Spring Boot + MyBatis + MySQL 作为可选同步层。本次迭代新增学习资料库和 AI 提炼能力，使应用从简单番茄任务工具升级为能够处理学习资料、生成任务、执行专注和复盘结果的完整学习工作流。

\section{展望}

后续可以从四个方向继续优化。第一，增加文件选择与 OCR 能力，让 PDF、图片和课件可以自动导入。第二，完善后端同步 UI，加入同步状态、冲突处理和多端备份。第三，增强 AI 功能，支持复习计划生成、错题整理、资料问答和周报生成。第四，补充真机测试、无障碍适配、通知可靠性和主题定制，使应用更接近可长期使用的成熟产品。
